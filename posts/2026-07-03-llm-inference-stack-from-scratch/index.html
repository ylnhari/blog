Rebuilding the LLM Inference Stack From Scratch in NumPy

By ylnhari · July 3, 2026 · llm, transformers, inference, numpy, from-scratch

If you deploy large language models for a living, you know the tools: vLLM, TensorRT-LLM, TGI. You know the words that make them fast — KV-cache, paged attention, speculative decoding, continuous batching. But knowing the words is not the same as knowing the machine. So I set out to build the whole thing from scratch, in pure NumPy, with one rule: every optimization has to come with a proof that it did not change the model's output.

The result is a small, readable library where you can trace every matrix and every gradient by hand — a trainable GPT and the five techniques that turn it into a serving engine. This post is the tour.

An optimization you cannot prove correct is a bug you have not found yet. Every step below ships with a numerical check that pins its output to the ground truth.

Step 1 — A GPT you can actually train, with backprop by hand

Most from-scratch transformer tutorials stop at the forward pass — they show you attention as a matrix operation and call it a day. That teaches the mechanism, not the model. The interesting part is learning, and learning is backpropagation.

So the first piece is a full decoder-only GPT — token and positional embeddings, pre-norm blocks, causal multi-head attention, an MLP, and a softmax cross-entropy head — with the backward pass written out by hand for every layer. No autograd. Each layer has a forward and a mirror-image backward; chain them in reverse and you have gradients.

Hand-derived gradients are easy to get subtly wrong, so before trusting them I check them numerically: nudge each parameter by a tiny amount, measure how the loss changes, and compare to the analytic gradient.

gradient check (max relative error): 1.55e-05
step    0  loss 2.6860
step  300  loss 0.0848
sample: 'to be or not to be that is the question. to be'

A relative error of ~1e-5 means the math is right. The loss falling from 2.69 (a random guess over the vocabulary) to under 0.1, and the model generating coherent text, means it learned. That gradient check is the template for everything that follows: prove it, then trust it.

Step 2 — The KV-cache: from O(n²) to O(n)

Naive autoregressive generation is wasteful. To produce token t, the model recomputes attention over the entire prefix — including the keys and values for tokens it already processed on every previous step. Those never change. Generating n tokens costs O(n²) when it should cost O(n).

The fix is the KV-cache: store each layer's keys and values, and on each step compute Q, K, and V for only the new token, attending against the cache. A subtle bonus falls out — you never build a causal mask, because the cache only ever holds the past.

Does it change the output? It must not. So I run a full forward and the step-by-step cached path and compare the logits at every position:

equivalence check (max |logit diff| cached vs full): 8.88e-15
naive    131.4 ms
cached    20.5 ms   (~6x faster, byte-identical text)

Identical to machine precision, several times faster. The cache changed the speed, not the answer.

Step 3 — RoPE: position as rotation

The learned positional embeddings in Step 1 have two problems: they cap context at the table size, and they encode absolute position when language mostly cares about relative position. Rotary Position Embeddings (RoPE) — used by Llama, Mistral, Qwen, and most modern open models — fix both with one elegant idea: instead of adding a position vector, rotate each query and key by an angle proportional to its position.

Because rotating q at position i and k at position j and then taking their dot product leaves a dependence only on the difference (i − j), the attention score becomes a pure function of relative distance. And since it is defined by a formula rather than a lookup table, it works at any position. The proof is that the score stays constant when you slide a fixed offset along absolute positions:

relative-position check (max spread at fixed offset): 9.33e-15
applied RoPE at position 100000 with no lookup table — OK

Step 4 — Paged attention and speculative decoding

Two ideas power modern serving, and both are provably output-preserving.

Paged KV-cache is vLLM's core trick. A contiguous cache per sequence fragments memory when you serve many requests of different lengths. Paging stores the cache in fixed-size blocks — like an operating system handing out memory pages — with a block table per sequence. The attention step still sees a normal contiguous view, so it is completely transparent:

paged-vs-contiguous cache (max |logit diff|): 0.00e+00

Exactly zero — paging changes where the keys and values live, not their values.

Speculative decoding attacks latency. A big model needs one forward pass per token and is memory-bound. So a small, cheap draft model proposes the next few tokens, and the big target model verifies all of them in a single forward. With greedy decoding you accept every proposed token that matches the target's own choice and stop at the first disagreement — which yields exactly the target's greedy output, using far fewer expensive forwards:

speculative output == target greedy: True
target forwards: greedy=60  speculative=27  (2.22x fewer)

Step 5 — Continuous batching: keep the batch full

The last piece is scheduling. Real requests have wildly different lengths — a reply might be 8 tokens or 200. Static batching runs a fixed group until its longest member finishes, so short requests leave their slots idle while everyone waits. Continuous batching treats the batch as rolling slots: the instant a sequence finishes, it is evicted and a waiting request takes its place. The batch stays full.

The scheduler changes when tokens are computed, never what they are — so every request's output must match generating it alone. On a workload with deliberately varied lengths:

every request's output == standalone greedy: True
static     : 71% slot utilization
continuous : 100% slot utilization  (1.41x fewer slot-steps for the same work)

Why build what you can pip install?

Because there is a difference between operating a system and understanding it — and that difference is exactly what separates a framework user from a framework builder. I deploy these systems in production; rebuilding them from first principles is how I learned to explain why they are fast, not just that they are.

The other reason is the discipline the whole project is built around: every single step is pinned to a numerical proof — a gradient check, an equivalence check, a utilization measurement. That habit — never trust an optimization you have not verified against ground truth — travels far beyond this toy code.

The full, heavily commented source, with a teaching notebook for each concept, is public here: github.com/ylnhari/learning. It is written to be read top to bottom and run cell by cell. If you have ever wanted to see the LLM inference stack with nothing hidden, clone it and turn the knobs — break the residual connections, remove the causal mask, shrink the draft model — and watch what happens. That is where understanding stops being words and becomes intuition.
