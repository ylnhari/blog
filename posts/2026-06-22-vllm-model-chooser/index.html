vLLM Model Chooser: Will This LLM Actually Fit on Your GPUs?

By ylnhari · June 22, 2026 (updated July 14, 2026) · vLLM, LLM, GPU, open-source, AI-tools

The Problem: Parameter Count Tells You Nothing

If you've ever tried to serve an LLM on your own GPUs, you've hit this wall: a model's parameter count tells you almost nothing about whether it will actually fit. A 70B model in BF16 wants ~140 GB of weights — but in FP8 it's ~70 GB, and in NVFP4 it squeezes into ~35 GB. That's three wildly different answers for the same model, and which of them you're allowed to use depends on whether your GPU has tensor cores for that format at all.

And weights are only half the story. The other half is the KV cache, which grows with context length and concurrency, and which vLLM greedily fills with whatever memory is left after the weights load. A model whose weights "fit" in 74 GB with 4 GB to spare doesn't fit in any useful sense — you have nowhere to put the conversation.

Introducing vLLM Model Chooser

vLLM Model Chooser is an interactive web tool that tells you which vLLM-compatible models actually run on your GPU setup — weights, KV cache, and format support together — with zero configuration and no installation.

  • 6 GPU types: L4, A100, H100, H200, B100, B200 — at any count from 1 to 8
  • 108 models with params, active params (for MoE), weight-only VRAM, context length and quantization variants
  • A real memory budget: physical VRAM × --gpu-memory-utilization − activation reserve = what's actually usable for weights + KV
  • KV-cache estimate from each model's real attention geometry — read from its HuggingFace config.json, sliding windows included
  • Quantization × GPU matrix: native tensor-core support vs. vLLM's software path, per architecture
  • Provenance on every number — the tool tells you which figures are measured-from-source and which are estimates

It's open-source under MIT: github.com/ylnhari/vllm-model-chooser

"Weights tell you whether the model loads. The KV cache tells you whether you can serve anyone."

The Memory Budget, the Way vLLM Actually Accounts For It

The heart of the tool is a memory budget that mirrors vLLM's own accounting rather than hand-waving at "VRAM needed". Open any model and you get the whole chain, itemized:

Take Llama-3.3-70B on 2× H100 80GB, in FP8, at 32K context:

  • 160 GB physical across the two cards
  • × 0.95 memory utilization → 152 GB (vLLM's own default is 0.90–0.92; 0.95 is the optimistic end, and it's a control you can change)
  • − 4 GB activation / CUDA-graph reserve → 148 GB usable
  • − 70 GB FP8 weights → 78 GB KV cache pool
  • ≈ 7 concurrent requests if every one of them fills the full 32K context

That last number is the one that changes decisions. "It fits" and "it fits, and you can serve seven simultaneous long-context requests" are different sentences, and only the second one is a deployment plan. (It's deliberately a floor: real traffic is mostly shorter than max context, and PagedAttention shares prefix blocks, so you'll usually do better.)

The KV Cache Is Computed, Not Guessed

Most VRAM calculators estimate the KV cache as some multiple of parameter count. That's wrong in both directions: it ignores grouped-query and multi-head-latent attention, which shrink the cache dramatically, and it over-counts MoE models, whose experts add parameters but no attention state.

So the Chooser doesn't guess. A generator script pulls each model's config.json from HuggingFace and computes bytes-per-token from the actual geometry — layers, KV heads, head dimension, and, crucially, sliding-window attention: layers that cap their cache at a fixed window instead of growing with context. That's why some models here show a startlingly small KV footprint. It isn't a bug; it's the architecture, and it's the same file vLLM itself reads at startup.

You can also switch the cache to FP8 (--kv-cache-dtype fp8), which halves it, and watch every model's fit verdict shift.

Every Number Says How Confident It Is

A tool that hands you a number with no provenance is asking you to trust arithmetic you can't see. So each figure carries a marker, and the page ends with a glossary that explains what each one means and what to do about it:

  • No marker — computed from the model's own published config.
  • ° — the model's repo is gated, so the attention geometry came from an ungated same-architecture mirror.
  • † — no config reachable anywhere; geometry hand-sourced from the model card or paper.
  • ‡ — no geometry at all, so the KV falls back to a coarse params×tokens proxy that ignores GQA/MLA and over-estimates MoE. The weakest number on the site, and it says so.
  • * — the activation reserve is an assumption. vLLM measures it by profiling a real forward pass at startup; the tool can only guess, so it lets you set it.

The honest headline: every VRAM and KV figure here is arithmetic, not a reading from a running server. Use it to shortlist hardware in seconds instead of hours — then confirm by actually starting vLLM.

Quantization: Native Silicon vs. vLLM's Software Path

vLLM supports BF16, FP8, INT8, INT4/AWQ/GPTQ, NVFP4, MXFP4 and MXFP8 — but "supported" hides an important distinction. Some formats run on native tensor cores; others are dequantized on the fly in software, which saves memory but gives you no speedup, and can be slower than a format the hardware understands.

  • BF16 — native everywhere, from Ada to Blackwell.
  • FP8 — native on Ada, Hopper and Blackwell; software (Marlin) on Ampere.
  • INT8 (W8A8) — native on every listed GPU; they're all Ampere or newer, with int8 tensor cores.
  • INT4 / AWQ / GPTQ (W4A16) — software on all GPUs: the weights are dequantized to FP16 for the actual math. Real memory saving, no int4 tensor-core speedup.
  • NVFP4 — Blackwell FP4 tensor cores only. Absent on L4, A100 and H100.
  • MXFP4 — native on Blackwell, software elsewhere (which is how gpt-oss-120b squeezes onto a single A100).
  • MXFP8 — native on Blackwell MX tensor cores only; unsupported on earlier hardware.

The tool never offers you a variant your GPU can't run, and a fit that only works through the software path is marked ✓* rather than a plain ✓ — because you should know what you're buying.

Multi-GPU: From One L4 to Eight B200s

Every model card shows fit at 1×, 2×, 4× and 8×, and the detail view walks all counts from 1 to 8, so questions like these get answered in a second:

  • "Can Llama-3.3-70B run on a single L4?" — No. 20.8 GB usable per card against 140 GB of BF16 weights.
  • "On one H100, then?" — Not really: FP8 weights are 70 GB against 74 GB usable, which leaves 4 GB of KV — a server that loads and then serves almost nobody. Two cards give you 78 GB of cache.
  • "What unlocks the 1T-parameter models?" — Eight B200s: 180 GB usable each, 1.4 TB total, enough for Kimi K2.7-Code or DeepSeek-V4-Pro in a 4-bit format.

How It's Built

The page is static and dependency-free at runtime — index.html plus two plain scripts, no bundler, no framework, no build step. Serve it with anything:

python3 -m http.server 5000
# then open http://localhost:5000

The data behind it is generated, not hand-written, which is the whole reason the numbers can be trusted. scripts/sync-data.mjs pulls the model inventory and per-model specs from vLLM's official recipe catalog (recipes.vllm.ai/models.json), cross-checks context length and attention geometry against each model's HuggingFace config.json, and derives weight-only VRAM from parameter count × bytes-per-parameter. A second script, npm run factcheck, audits the result back against the recipes, and a Node test suite covers the fit logic. Node is only needed to regenerate or test — never to view.

The quantization × architecture matrix comes from vLLM's own supported-hardware docs, corroborated per-model against the hardware hints in each recipe. When a blog post disagrees with a recipe, the recipe wins.

Conclusion

Sizing an LLM deployment is arithmetic that nobody enjoys doing twice: weights, format support, KV geometry, memory utilization, activation reserve, concurrency. The Chooser does it for 108 models across six GPUs and eight counts, shows its working, and is candid about which numbers are estimates.

Pick your GPU, turn on the KV estimate at the context length you actually serve, and see what survives.
